You are reviewing a set of exploration/visualization tools built for the CLAWMARKS project, a
LoRA (image-model fine-tune) hyperparameter and prompt search being written up as a whitepaper.

## Background

The project runs a MAP-Elites-style quality-diversity search over image generation settings.
Every generated image is scored on two axes via DINOv2 embeddings: "faithfulness" (cosine
similarity to the centroid of ~31 real training images) and "novelty" (1 minus similarity to the
nearest neighbor in an "already explored" reference set). The search runs in generations,
alternating "explore" (new subjects/settings) and "exploit" (mutate near known-good images).
Round 1 produced 3392 images; round 2 (explore-heavy, GPT-5.5-seeded subjects, penalized for
resembling round 1's output) just finished with 280 more.

A previous brainstorm with a model called Fable produced 8 tool ideas to help a human explore
individual images/clusters and understand the shape of the solution-space map being built. All
8 have now been built, at varying depth:

1. `notes/build_solution_map.py` + `notes/build_map_view.py` -> `map.html`: re-embeds every
   image with DINOv2, fits UMAP on the joint real+generated embedding space, renders an
   interactive canvas scatter with a generation slider/play control and a "nearest real image"
   mode-collapse bar chart.
2. (folded into map.html's generation slider)
3. `notes/build_coverage_map.py` -> `coverage.html`: an 8x8 faithfulness x novelty density
   heatmap, flags "frontier" cells (empty but adjacent to a dense cell), click a cell to preview
   its top image or open a modal showing every image in that cell.
4. `notes/build_redundancy_view.py` -> `redundancy.html`: connected-components clustering over
   precomputed top-16 DINOv2 nearest-neighbor edges (with cosine scores), adjustable similarity
   threshold slider, shows near-duplicate clusters.
5. `notes/build_elite_archive.py` -> `archive.html`: one image per occupied faithfulness x
   novelty cell (human pick preferred, else highest novelty) -- the actual MAP-Elites archive.
6. (folded into map.html's nearest-real-image chart)
7. `notes/build_lineage_view.py` -> `lineage.html`: exploit-chain parent/child tree. Currently a
   placeholder -- no image in the dataset has parent-tracking data yet (a patch just landed in
   `notes/run_uncanny_allnight2.py` to start recording it, but only takes effect on the *next*
   process restart).
8. `notes/build_novelty_decay.py` -> `novelty_decay.html`: per-prompt-family mean novelty across
   generations as sparkline charts, sorted worst-trending first, to flag exhausted prompts.

Plus the pre-existing `notes/build_scan_gallery.py` -> `scan.html` (the full sortable/
filterable/searchable gallery with a lightbox, DINOv2 "show similar" browsing, and a human
"pick as winner" curation control that feeds round 2's exploit pool via
`notes/uncanny_sweep/user_picks.json`), and `notes/curation_server.py` (the static file + pick
API server all of this is served through, on port 8420, with `scan.html` as the default page
other tools link back to).

All pages are static HTML/JS generated by Python build scripts reading precomputed JSON data
(no external CDN dependencies, no heavy backend). Every image thumbnail across every tool is
meant to open the *same* scan.html lightbox via a `scan.html?open=<tag>` deep link
(`window.open(...)`), rather than each tool reimplementing its own image viewer.

## Your task

You have full read access to the `notes/` directory (and the rest of the repo) via `--dir`.
Actually read the Python build scripts and, if useful, the HTML they emit (run the build
scripts yourself if you want to see real output -- they're idempotent and safe to re-run,
though `build_solution_map.py` takes ~28 minutes to re-embed everything from scratch, so avoid
re-running that one; everything else finishes in seconds).

Do two things:

### Part A: Full review of the existing tools

Look for real problems, not style nits: broken or inconsistent deep-linking, JS bugs, a chart
that would mislead someone about the actual data (e.g. binning that hides more than it shows,
a stat computed wrong), performance problems (are we about to repeat the mistake where
full-resolution PNGs got used as thumbnails and made scan.html laggy?), and inconsistencies
between tools (e.g. different pages using different bin counts or scoring conventions without
explanation). Call out anything that's actually broken or actively misleading first, then lower-
priority polish.

The team wants to keep going. Suggest concrete next tools or improvements to what exists,
specifically ones that would change how the team decides where to explore next, which images/
regions are genuinely redundant, or how well the search is doing -- not just "make it prettier."
Consider: round 2's 280 images aren't merged into any of these tools yet (they live in a
separate `notes/uncanny_sweep2/` directory with their own `gallery.html`) -- is merging round 1
and round 2 into one unified view worth prioritizing, and how would you do it given the
directory split? Also consider whether the lineage-tree placeholder (idea 7) is worth
backfilling for round 1's data somehow, or genuinely has to wait for fresh data.
