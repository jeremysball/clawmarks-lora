# Part A Findings

1. **`map.html` is linked but not present.** `notes/build_explore_hub.py:14` links the hub to `map.html`, and `notes/build_scan_gallery.py:174` offers the same target in the tool menu, but `notes/uncanny_sweep/map.html` and `notes/uncanny_sweep/solution_map_data.json` are absent. A user following the main navigation hits a 404 for one of the headline tools. Either run `notes/build_solution_map.py` then `notes/build_map_view.py`, or hide the link until the expensive UMAP data exists.

2. **Round 2 is invisible to every new tool.** `notes/build_scan_gallery.py:19`, `notes/build_coverage_map.py:16`, `notes/build_elite_archive.py:17`, `notes/build_redundancy_view.py:20`, `notes/build_novelty_decay.py:13`, and `notes/curation_server.py:25` all hard-code `notes/uncanny_sweep`. `notes/uncanny_sweep2/scored_manifest.json` contains 280 scored images, but none of these pages read it, so the tools answer "what happened in round 1" while the current research question is whether round 2 escaped the plateau. This is not only incomplete; it can make the search look stalled or redundant after new data exists.

3. **Round 1 and round 2 novelty scores are not directly comparable as stored.** Round 1 computes novelty as `1 - nearest real image similarity` in `notes/build_uncanny_gallery.py:120-126`. Round 2 computes novelty as `1 - max(nearest real, nearest round-1 output)` in `notes/run_uncanny_allnight2.py:251-258`. A unified heatmap that bins the existing `novelty` field would mix two different meanings: absolute distance from the real training set versus distance from both the real set and prior generated images. Store `novelty_real` and `novelty_global`, then let each chart state which one it uses.

4. **The coverage map describes density on quantile bins, not fixed metric space.** `notes/build_coverage_map.py:26-31` derives faithfulness and novelty edges from the same manifest it plots. That makes each axis roughly equal-populated by construction, so high-count cells mostly show correlation structure, not absolute density in score space. The page text at `notes/build_coverage_map.py:135-139` says it is the same faithfulness x novelty plane as `gallery.html`, but the actual bin edges shift whenever the manifest changes. This will especially mislead after round 2 is merged unless the tool uses fixed, documented score edges or a toggle between quantile and absolute bins.

5. **The lineage renderer will drop children whose parent is outside the current manifest.** `notes/build_lineage_view.py:53-58` records children with missing parents in `roots`, but `roots` is never rendered. `notes/build_lineage_view.py:66-68` builds `tree_html` only from parents present in `children_by_parent`, so a round-2 exploit whose `parent_tag` points to a round-1 user pick can disappear if the page reads only `uncanny_sweep2`. This matters because `notes/run_uncanny_allnight2.py:178-182` records exactly that kind of parent tag.

6. **The redundancy view can overstate duplication at low thresholds.** `notes/build_redundancy_view.py:85-99` clusters by connected components over top-16 neighbor edges. Connected components are transitive: A can connect to B, B to C, and C to D even if A and D are not near duplicates. The summary at `notes/build_redundancy_view.py:108-111` calls these "effective clusters," which is fair at strict thresholds but misleading at loose ones unless the page also reports minimum internal edge score or warns about chaining.

7. **The original binned atlas still embeds thumbnails as base64.** `notes/build_uncanny_gallery.py:74-80` creates data URIs, and `notes/build_uncanny_gallery.py:184-188` embeds up to 12 thumbnails per occupied cell. The newer scan tool fixed this with `notes/build_thumbnails.py`, but `gallery.html` can still become heavy as the run grows. This is lower priority than the broken map and round-2 merge, but it repeats the known full-page weight problem in miniature.

# Part B Ideas

1. **Build a unified manifest layer first.** Create a small script that reads both `uncanny_sweep/scored_manifest.json` and `uncanny_sweep2/scored_manifest.json`, prefixes or records `round`, rewrites thumbnail paths relative to one served root, and keeps both novelty definitions. This is the highest-priority improvement because every downstream tool becomes more honest once it can compare rounds in one place. Difficulty: medium, mostly path hygiene and one careful data schema decision.

2. **Add a round-delta dashboard.** Show round 1 versus round 2 on the same fixed faithfulness x novelty grid, with counts, best images, and per-cell gains or losses. This directly answers whether the explore-heavy run reached new regions or only repopulated old bins. Difficulty: medium after the unified manifest exists.

3. **Add a novelty-definition toggle.** Let coverage, archive, and scan switch between distance from real images and distance from real plus prior generated images. The first says "does this still sit near the real style," while the second says "did this escape what we already explored." Difficulty: low to medium if the unified scorer persists both columns.

4. **Replace archive selection with a Pareto and pick-aware cell view.** For each cell, show the highest novelty image, highest faithfulness image, and any human pick instead of one winner. This prevents one noisy novelty outlier from hiding a more useful exploit seed. Difficulty: low, mostly HTML layout and a slightly richer cell JSON.

5. **Backfill approximate lineage for round 1 only as provenance, not fact.** True parent tracking cannot be reconstructed because `notes/run_uncanny_allnight.py` never recorded which elite `random.choice` selected. A useful fallback is to infer likely parents for exploit images by nearest earlier-generation neighbor, label it "inferred by DINOv2," and keep recorded `parent_tag` separate. Difficulty: medium, and it should never feed the whitepaper as ground truth.

6. **Add a cluster health report.** For each redundancy cluster, report size, prompt mix, generation span, best novelty, and whether later generations keep adding near-duplicates. This would show which regions to retire from exploit and which clusters still improve. Difficulty: medium, using existing similarity edges plus manifest metadata.

7. **Add a prompt allocation recommender.** Combine novelty decay, per-prompt faithfulness, and redundancy to suggest "retire," "keep exploring," or "exploit carefully" for each prompt family. This turns the charts into an action list for the next generation. Difficulty: medium, with simple rules first and no model needed.
